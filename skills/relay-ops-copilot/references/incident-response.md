# Incident Response

## 1. 事故等级

### P1

- 主 API 域名不可用
- 大面积 key 请求失败
- 面板和 API 同时不可用
- 出现明显泄露风险

### P2

- 某条主卖路由不可用
- 高质量路由持续失败
- 错误率明显升高
- 自动补给链路不可用

### P3

- 某些模型临时不可用
- 某些上游有回落或价格异常
- 某个非主卖商品不能正常使用

## 2. 处理顺序

1. 先确认影响面：域名、路由、模型、用户范围。
2. 先止血：隐藏故障路由、暂停售卖、切换到健康备选路由。
3. 再定位：看日志、看上游响应、看模型返回是否与请求一致。
4. 最后复盘：形成日报和后续预案。

## 3. 推荐动作

### 路由不可用

- 暂停展示该路由
- 保留已发 key，不做删除
- 给出健康备选路由

### 模型回落

- 标记为假通
- 下调该商品宣传口径
- 必要时从可售清单移除

### 自动补给异常

- 改回激活码优先
- 暂停自动签发新 key
- 保留现网 key 的可见性和说明

## 4. 报告格式

每次事故输出：

- 发生时间
- 影响范围
- 证据
- 当前状态
- 已采取动作
- 建议下一步
