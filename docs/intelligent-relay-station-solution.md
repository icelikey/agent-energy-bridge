# 智能中转站解决方案

## 1. 目标定位

这套方案不是把中转站继续做成“只有 Key 分发和转发”的工具，而是把它升级成一套面向 Agent 生态的服务系统：

- 对终端用户：提供稳定接入、预算护栏、自动加油、模型推荐、接入文档
- 对代理商：提供商品分层、激活码补给、资料包、FAQ、交付模板
- 对站长：提供异常巡检、路由体检、日报、修复建议、影子变更建议
- 对开源生态：提供可插拔 adapter 和通用 skill，不绑死某一个中转站

一句话总结：

`智能中转站 = 网关 + bridge + skill + 智能运维 + 内容分发能力`

## 2. 总体架构

### 2.1 四层分工

1. 网关层
   - `sub2api / new-api / one-api compatible`
   - 负责真实转发、路由、分组、Key、计费、并发控制、上游配置

2. Bridge 层
   - `agent-energy-bridge`
   - 负责预算护栏、模型能力测评、模型推荐、补给编排、接入文档渲染、兼容保护

3. Skill 层
   - `token-energy-station`
   - `relay-ops-copilot`
   - `relay-growth-playbook`
   - 负责把 bridge 的能力变成智能体可调用的工作流

4. 自动化运维层
   - heartbeat / cron / 每日巡检 / 异常告警 / 日报
   - 负责健康检查、线路体检、异常汇总、回切建议、日报归档

### 2.2 三个 skill 的角色

#### `token-energy-station`

负责“加油和路由”。

适合场景：
- 余额不足
- token 消耗过快
- 需要推荐模型
- 需要补给额度
- 需要签发新 key 或输出接入文档

#### `relay-ops-copilot`

负责“智能运维”。

适合场景：
- 检查中转站是否健康
- 检查上游和路由哪些可用
- 分析 503 / 400 / 无可用渠道 / 回落模型等异常
- 输出日报、事故等级、修复建议
- 做影子切换建议，不直接影响现网 key

#### `relay-growth-playbook`

负责“干货输出和增长材料”。

适合场景：
- 设计高质量版、混合版、全能版商品结构
- 给代理商生成资料包、FAQ、发卡说明、接入教程
- 给闲鱼、社群、海报、教程生成文案骨架
- 统一终端用户、代理商、合作智能体三类材料口径

## 3. 关键设计原则

### 3.1 无损接入

- 不覆盖已发出去的 key
- 不重命名现网已售路由
- 不直接替换线上模型映射
- 新能力先做影子建议，再由人工确认启用

### 3.2 域名优先

- 对外只展示域名
- 复制按钮、文档模板、教程示例都不暴露服务器 IP
- 代理资料、截图、脚本示例统一用域名占位

### 3.3 预算护栏默认开启

- 默认有单日预算
- 默认有小时 token 上限
- 默认有自动补给次数上限
- 默认对高价模型做白名单或显式批准

### 3.4 商品和技术解耦

- 网关层只负责“能否调通”
- bridge 层负责“值不值得调、该不该补、该推荐什么”
- growth skill 负责“怎么卖、怎么讲、怎么交付”

## 4. 推荐产品结构

### 4.1 高质量版

- 适合：高难编码、复杂修复、长链路推理
- 入口：高质量专属路由
- 特点：高质量、高成本、严格预算
- 典型模型：高质量 Claude / O3 / GPT-Codex

### 4.2 混合版

- 适合：日常办公、普通对话、轻分析、一般自动化
- 入口：混合通用路由
- 特点：成本友好，优先性价比模型
- 典型模型：Claude 4.6 mixed / Kimi / Minimax / 其他成本更优模型

### 4.3 全能版

- 适合：新手、代理统一交付、跨客户端接入
- 入口：全能渠道
- 特点：一个入口适配多协议、多模型风格
- 逻辑：先按协议分流，再按任务类型和预算层级分流

## 5. 数据对象

建议 sidecar 或 bridge 持有以下逻辑对象：

- `route_registry`
  - 路由名、协议、质量等级、预算等级、状态、当前建议动作
- `model_registry`
  - 模型能力、成本层、质量层、稳定性、路由归属
- `budget_policies`
  - 每个账号、Key、代理、套餐的预算护栏
- `activation_codes`
  - 金额、状态、有效期、适用对象、创建人
- `reseller_profiles`
  - 代理等级、倍率、可售商品、可查看客户范围
- `report_snapshots`
  - 每日报表、异常统计、修复建议、成功率和失败率变化

## 6. 自动化工作流

### 6.1 用户加油工作流

1. `token-energy-station` 读取余额和 usage
2. 判断是否触发预算护栏
3. 低余额优先尝试激活码
4. 无激活码时判断是否允许自动补给或签发新 key
5. 输出：可用额度、建议模型、路由、接入文档、风险提示

### 6.2 智能运维工作流

1. `relay-ops-copilot` 检查域名、健康接口、容器、日志、上游可达性
2. 对每条关键路由做最小化冒烟测试
3. 标记状态为：`available / degraded / unavailable / fake-available`
4. 输出：异常列表、根因猜测、优先级、修复建议、是否建议临时下线
5. 汇总到日报，并形成下一轮巡检基线

### 6.3 干货输出工作流

1. `relay-growth-playbook` 读取当前商品结构、模型目录、价格策略
2. 按对象拆分材料：终端用户 / 代理商 / 合作智能体
3. 输出接入文档、FAQ、发卡说明、话术、商品对比表
4. 对外口径自动遵守域名优先和预算护栏原则

## 7. 智能运维能力清单

必须具备的能力：

- 站点域名可用性检查
- 本地 `/health` 与容器健康检查
- 最近错误日志采样
- 路由级冒烟测试
- 上游模型列表和请求返回对比
- “请求 4.7 实际回 4.6” 这种回落行为识别
- 日报输出
- 影子变更建议

建议逐步增加：

- 自动冻结连续失败路由
- 自动切回健康备选路由
- 高失败率告警
- 高价模型消耗过快预警
- 异常客户行为识别

## 8. 干货 skill 输出物

`relay-growth-playbook` 建议稳定输出以下模板：

- 新手接入说明
- 代理商快速上手手册
- 模型与接口目录
- 充值与激活码 FAQ
- 高质量版 / 混合版 / 全能版对比表
- 闲鱼商品标题骨架
- 商品卖点说明
- 微信沟通话术
- 风险提示模板

## 9. 与现有代码的映射

### 9.1 已有能力

当前 `agent-energy-bridge` 已经覆盖：

- `BudgetGuard`
- `CompatibilityGuard`
- `EnergyEngine`
- `ModelCapabilityBenchmark`
- `ModelSelector`
- `RefuelOrchestrator`
- `ReferralEngine`
- `GenericOpenAIGatewayAdapter`
- `token-energy-station`

### 9.2 还需补强的能力

为了形成完整的“智能中转站服务方案”，接下来建议补：

- `relay-ops-copilot` 的真实运维脚本或 sidecar 接口
- 路由巡检结果持久化
- 日报聚合器
- 商品与代理资料模板渲染器
- 闲鱼 / 发卡平台 webhook adapter
- 路由健康度与模型回落行为的历史趋势表

## 10. 实施顺序

### P0：先稳定现网

- 只保留真实可卖的路由
- 把无效路线移出展示面
- 固定对外域名与复制模板
- 给现网建立日报基线

### P1：补 sidecar

- 提供 `balance / usage / redeem / issueKey / docs / recommend` 接口
- 让 `token-energy-station` 真正可调用 sidecar

### P2：补智能运维

- 落地 `relay-ops-copilot`
- 接入 heartbeat / cron
- 生成日报和异常建议

### P3：补干货与增长

- 落地 `relay-growth-playbook`
- 统一输出代理资料、终端文档、售卖模板
- 对接闲鱼和发卡平台的自动交付动作

## 11. 最终形态

最终这套系统不只是“卖 API Key”，而是：

- 能判断该用什么模型
- 能判断预算是否安全
- 能在余额不足时自动加油
- 能对异常线路做巡检和预案
- 能自动给下游和代理发完整资料
- 能把中转站变成一个可运营、可扩展、可复制的智能基础设施
