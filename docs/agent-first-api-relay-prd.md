# Agent-First API Relay PRD

## 1. 产品定义

这不是一个普通的 API 中转站。

它的核心目标不是“把请求转出去”，而是：

- 帮智能体持续续航
- 帮智能体自动选择更合适的模型
- 帮智能体在保证质量的前提下节约 token 和成本
- 帮站长把中转站做成可运营、可控风险、可扩展的智能基础设施

一句话定义：

`这是一个专门为 AI 智能体服务的 API 中转站，负责供能、调度、节流、补给和优化。`

## 2. 和普通中转站的区别

### 普通中转站

- 重点是转发
- 提供 Key
- 提供基础计费
- 用户自己决定用什么模型
- 用户自己承担预算失控和模型选择错误

### 智能体中转站

- 不只是转发，还负责预算控制和任务调度
- 面向的不是普通人，而是 Claude Code、Codex、OpenClaw、Harness 一类 Agent
- 会根据任务自动推荐模型、推荐路由、限制风险
- 余额不足时可以自动补给或激活码补给
- 能识别高成本、高失败率、假通路由、模型回落
- 能输出给 Agent 可直接消费的接入文档和操作说明

## 3. 核心价值

### 3.1 给智能体

- 自动选择更划算的模型
- 自动判断是否值得调用高价模型
- token 不足时自动加油
- 在预算快超限时自动降级
- 用统一入口接不同协议和模型

### 3.2 给用户

- 不需要研究模型差异
- 不容易因为乱用高价模型而爆成本
- 一个地址、一套文档、一种充值逻辑就能持续用

### 3.3 给站长

- 可以把“预算护栏 + 自动补给 + 模型推荐 + 运维”组合成更高价值服务
- 可以形成高质量版、混合版、全能版等可运营商品
- 可以对代理、终端用户、合作智能体输出不同层级的服务能力

## 4. 核心能力模型

这套系统建议拆成五个核心模块。

### 4.1 Fuel Engine 供能引擎

负责：

- 查询余额
- 查询 token 消耗
- 激活码充值
- 自动补给
- 签发新 key
- 输出接入文档

目标：

- 让 Agent 不因为额度不足而中断
- 让续航能力自动化

### 4.2 Budget Guard 预算护栏

负责：

- 每日预算
- 每小时 token 限制
- 高价模型限制
- 自动补给次数上限
- 自动补给金额上限

目标：

- 防止单个 Agent 失控
- 把风险控制从人工变成系统默认行为

### 4.3 Model Router 模型调度器

负责：

- 按任务类型选择模型
- 按协议选择路由
- 按预算选择高质量版或混合版
- 支持全能渠道统一入口
- 多任务工作流下分步骤推荐模型

目标：

- 让“贵模型只在该贵的时候贵”
- 在质量和成本之间自动做平衡

### 4.4 Efficiency Brain 节能优化器

负责：

- 统计不同模型的单位 token 价值
- 记录成功率、首轮成功率、时延、失败率、回落情况
- 形成能量评分
- 对同类任务持续推荐更高性价比模型

目标：

- 不是只节流，而是提高每 1 个 token 的产出价值
- 让整个站越用越聪明

### 4.5 Ops Copilot 智能运维副驾

负责：

- 健康检查
- 上游可用性检查
- 路由冒烟测试
- 错误日志分析
- 日报生成
- 假通识别
- 修复建议和回切预案

目标：

- 让中转站对 Agent 来说持续可用
- 降低“站点看着在线但其实不可卖”的风险

## 5. 最重要的产品特性

### 特性一：专门为智能体设计

默认服务对象是：

- Claude Code
- Codex
- OpenClaw
- Harness
- 其他支持 OpenAI / Anthropic / Google 风格协议的 Agent

系统输出的文档和接口都要适合这些智能体直接使用，而不是只适合人类手工阅读。

### 特性二：优先考虑续航

当余额不足时，系统不要只报错，而应该：

1. 先检查是否有可用激活码
2. 没有激活码时判断是否允许自动补给
3. 如果预算不允许，自动降级到更便宜的路由
4. 仍然不行时，输出明确的加油指令和最小可用方案

### 特性三：优先考虑 token 价值

不是单纯“越便宜越好”，而是：

- 高难任务时优先成功率
- 日常任务时优先成本效率
- 多轮工作流时优先稳定性
- 对重复任务优先推荐历史表现更优的模型

### 特性四：兼容保护

- 不影响已开通 key
- 不影响已售路由
- 新能力先影子接入
- 不因为加了智能层把老客户用挂

## 6. 三种标准商品形态

### 高质量版

- 面向重度编码和复杂任务 Agent
- 优先高质量、高稳定性
- 价格更高
- 预算限制更严

### 混合版

- 面向普通智能体任务
- 优先性价比
- 在质量足够时尽量省 token 成本

### 全能版

- 一个入口兼容多协议
- 适合新手和代理分发
- 适合多模型、多客户端混合使用

## 7. 一个完整调用闭环

一个理想的 Agent 使用流程应该是：

1. 智能体发起任务
2. 中转站判断任务类型
3. 推荐模型和路由
4. 调用前检查预算和余额
5. 余额不足则自动补给或激活码补给
6. 执行请求
7. 记录 token、成本、成功率、时延
8. 更新模型能量评分
9. 下一次推荐更优选择

这才是智能体中转站的闭环。

## 8. 最小可行版本 MVP

第一阶段只需要做四件事：

1. 统一接入入口
   - 支持 OpenAI / Anthropic 风格

2. 预算护栏
   - 日预算、小时 token、自动补给限制

3. 自动加油
   - 余额查询、激活码充值、低余额补给、接入文档输出

4. 模型推荐
   - 高质量版、混合版、全能版三层推荐

如果这四件事跑通，这个站就已经不是普通中转站了。

## 9. 第二阶段增强

第二阶段建议增强：

- 路由级健康度评分
- 模型能力测评历史库
- 每日运营日报
- 假通和模型回落识别
- 代理商资料自动生成
- 发卡平台自动交付

## 10. 最终系统口号

可以把它理解成：

- 给智能体加油
- 给智能体选路
- 给智能体省钱
- 给智能体兜底

最终不是卖一个 API 地址，而是提供：

`Agent Runtime Infrastructure as a Service`

也就是：

`面向智能体的运行时基础设施服务。`
